# PRD — 필라테스 시퀀스 생성 앱 (`pilaiv2`)

> 상태: **v0.7 초안** (대화로 채우는 중)
> ⚠️ = 핵심 리스크 / ❓ = 확정 필요

## 1. 한 줄 정의

필라테스 선생님이 **회원 맞춤 운동 "시퀀스"를 빠르고 안전하게** 만들도록 돕는 LLM 기반 앱.

## 2. 문제 — 왜 만드나

- **시퀀스(수업 운동 구성) 짜기가 선생님의 핵심 노동이자 고통.** 회원마다 "어디가 안 좋은지 / 뭘 개선할지"를 타겟해 동작을 골라야 하는데, 그러려면 끝없이 공부해야 함.
- 자격증·커리큘럼 수준이 중구난방. BASI처럼 틀이 잡힌 곳도 경력이 쌓이면 시퀀스 만들기에서 막힘.
- 그래서 선생님들이 **돈을 계속 쓴다**: 현장 강의, 유튜브 시퀀스 구독, 시퀀스 문서 모음(수십만 원) 구매.
- 기존 해결책은 **정적**이다. 고정된 문서라 회원 개별 상태에 맞춰주지 못함.

→ **지불 의사가 검증된 시장.** 빈자리는 "회원 맞춤 + 동적 생성".

## 🏰 제품 해자 — 왜 그냥 ChatGPT가 아닌가

> **검증됨:** 아내(현직 선생님)가 이미 LLM에 직접 프롬프트를 넣어 시퀀스를 뽑아봤고, 프롬프트가 서툴러도 "웬만큼 나온다"고 평가. → §6 파라메트릭 가정은 실전 확인됨.

"생성"은 누구나 ChatGPT로 한다. 이 앱의 가치는 생성이 아니라 **LLM을 프로덕션에서 신뢰 가능하게 만드는 시스템**이다:

1. **신뢰 (안전 검증)** — 혼자 ChatGPT 쓰면 틀린 금기를 거를 안전망이 없다. §9 verifier가 잡는다.
2. **맥락 누적** — ChatGPT는 회원을 모른다. 이 앱은 회원 상태·이력·지난 시퀀스를 기억하고 반영(Phase 2+).
3. **워크플로우** — 프롬프트 노가다 없이: 폼 입력 → 구조화된 시퀀스 → 편집 → 저장.
4. **집단 학습** — 한 선생님의 편집(diff)이 전체 시스템을 개선(§8 플라이휠). 혼자 쓰는 ChatGPT엔 없는 복리.

**핵심:** "프롬프트 서툴러도 웬만큼"은 뒤집으면 "사용자마다 들쭉날쭉"이다. 도메인 전문가가 아닌 선생님은 틀린 결과를 걸러낼 수 없다. **그 변동성을 시스템이 흡수해 누가 써도 일관된 안전·품질을 보장**하는 것이 제품의 본질.

## 3. 타깃 사용자

- **1차 (이번 목표): 필라테스 선생님.**
- 검증자: 개발자 본인의 아내가 현직 필라테스 선생님 → 도메인 검증 가능.
- 이후 Phase: 회원(수강생).

## 4. 제품 로드맵 (Phase)

| Phase | 내용 |
|-------|------|
| **Phase 1 (MVP)** | 시퀀스 생성기. 기본 시퀀스 생성 + 회원 컨디션(예: 목 통증) 반영해 **금기 동작 제외/대체**. + **선생님 편집/피드백 캡처**(플라이휠 시드). |
| **Phase 2** | 회원 상태·일정·운동 이력 관리. 근육 활성도 등을 input으로 **커스터마이징된 시퀀스** 생성. |
| **Phase 3** | 회원용 앱. 회원이 일정 / 컨디션 체크 / 운동 이력 / 근육 활성화 직접 관리. |

## 5. 시퀀스 = 핵심 도메인 객체

**입력 — 2계층 (Phase 1 확정):**
- **회원 프로필 (영속, 1회 등록):** 이름, 나이/성별, 만성 제약·병력(예: 목 디스크), 장기 목표.
- **그날 세션 입력 (매 수업):** 오늘 컨디션/새 통증, 이번 수업 목표, 사용할 기구, 수업 길이.
- → 만성 제약(프로필) + 오늘 컨디션(세션) **둘 다** 안전 검증(§9)의 입력.

**출력 (시퀀스):** 순서가 있는 동작 리스트. 각 동작 =
`{ 기구, 동작명, 반복수, 시간, 타깃 근육, 난이도 }`

**흐름 제약 (단순 나열이 아님 — 생성 원칙):**
- **구조:** 웜업 → 본운동 → 쿨다운.
- **기구 블록화:** 기구 전환 최소화 (리포머 묶음 → 매트 묶음 → 체어 묶음). 막 섞으면 부자연스러움.
- **강도 곡선 / 근육군 밸런스:** 강도의 오르내림과 부위 균형을 고려.

**Phase 1 데이터 모델 (개념):**
- `Member` (영속): 이름, 나이, 성별, 만성 제약[], 장기 목표[].
- `Session` (수업 1회): `member_id`, 그날 컨디션·통증[], 그날 목표, 기구[], 길이, **생성본**, **최종본**(편집 후), verifier 결과.
- 생성본 ↔ 최종본 **diff = 플라이휠 데이터**(§8). 회원·컨디션에 묶여 축적.

## 6. 기술 방향 & 데이터 전략

- **v1:** ML 추천 알고리즘 → 성능 미달 (데이터 편중·부족).
- **v2 (현재):** LLM 기반. Claude 활용.
- **데이터 전략 — RAG 풀구축 안 함** (데이터 편중·비공개·학습 난이도):
  - **동작 생성·흐름 구성** → LLM 파라메트릭 지식으로 생성.
  - **금기/안전 매핑** → 파라메트릭에만 의존하지 않고 **검증된 얇은 안전 가드**(아내가 큐레이션한 "위험 조합" 룰)로 보강.
  - **RAG는 후순위** — eval에서 반복적으로 약한 영역이 드러나면 그 부분만 타깃 도입.
- ⚙️ **비용·지연 = 1급 설계 제약.** `gen→verify→repair`는 곧 LLM 호출 횟수. 룰 우선 검증·모델 티어링·프롬프트 캐싱·repair 상한으로 관리. 비용·지연도 eval로 추적(§7). 상세 레버는 `ARCHITECTURE.md`.

## 7. Harness 전략 (이 프로젝트의 엔진)

- 아내(도메인 전문가)의 **"맞다 / 틀렸다 / 위험하다" 판단을 골든 eval 셋으로 코드화.**
- 모델·프롬프트를 바꿀 때마다 이 eval로 **회귀 검증**.
- **안전 회귀(금기 위반)는 hard fail.**
- eval이 쌓일수록 신뢰도 상승 = harness engineering의 핵심 루프.

## 8. 피드백 루프 (데이터 플라이휠)

> 전제: 처음부터 완벽한 출력은 불가능하다. **선생님의 사용 과정 자체가 학습 데이터를 만든다.**

**신호 (값진 순서):**
- **편집 diff** — AI 생성본 vs 선생님 최종본. "틀린 지점 + 정답"을 동시에 알려주는 **가장 값진 신호.** eval 케이스 / few-shot 예시 / (먼 미래) fine-tuning 데이터로 재사용.
- 명시적 평가 — "쓸만함 / 안 맞음 / 위험함".
- 채택·거부 여부.

**라우팅 (피드백이 흐르는 3경로):**
- 생성 품질 문제 → **프롬프트·few-shot** 개선
- 사실·금기 오류 → **지식·안전 룰** 확장
- 반복되는 실수 → **eval harness** 확장 (§7)

**진화 = "harness system upgrade"의 실체:**
- 초기: 개발자 + 아내가 피드백을 보고 **수동으로** 프롬프트·룰 갱신.
- 이후: 피드백을 묶어 **반자동**(few-shot 후보 자동 추출, 약점 영역 자동 탐지)으로 격상.

⚠️ **설계 함의 — MVP 필수:** "완벽한 출력"은 불가능해도 **"완벽한 캡처 장치"는 1일차부터 가능.** MVP부터 **생성본/편집본 diff를 반드시 저장**한다. 나중에 붙이면 가장 비싼 초기 사용자 데이터를 영구 손실.

## 9. 생성-검증 파이프라인 (런타임 구조 개요)

> 상세 설계는 추후 `ARCHITECTURE.md`. 여기선 핵심 구조만.

```
Generator → Verifier(룰 + critic) → [실패] repair 재생성(최대 N회) → 통과 → 선생님
                                       └ 안전 위반 = hard block (절대 통과 X)
```

- **Generator** — 시퀀스 생성 (파라메트릭 + §5 흐름 규칙).
- **Verifier — 두 층:**
  - **결정론적 체크 (코드/룰):** 기구 제약 준수, 시간 합, **금기 블랙리스트**, 웜업/쿨다운 구조. **안전은 이 층이 책임진다.**
  - **LLM critic:** 흐름 자연스러움, 강도 곡선, 목표 부합 등 룰로 못 잡는 미묘한 판단.
- **Repair 루프** — verify 실패 시 재생성/수정(최대 N회). **안전 위반은 hard block.**

⚠️ **함정 — LLM이 LLM을 검증하는 맹점:** generator와 critic이 같은 LLM이면 **같은 맹점을 공유**한다. 흐름·완성도는 LLM critic이 잘 잡지만, **사실·금기는 독립 ground truth(룰/데이터) 없이는 못 잡는다.** → 안전 판정에 LLM-judge를 쓰지 않는다.

**harness와 공유:** 런타임 결정론적 체크 = §7 오프라인 eval의 채점 기준. **한 벌로 양쪽에서 사용.** verifier가 런타임에 잡은 실패도 §8 플라이휠로 환류.

## ⚠️ 핵심 리스크

- **환각 → 부상.** LLM이 금기/운동을 그럴듯하게 지어내면 회원 안전 위협, 선생님 신뢰 상실(재사용 안 함).
- 방어: §6 안전 가드 + §7 eval harness.
